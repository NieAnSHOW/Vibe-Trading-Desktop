## Why

AgentLoop 已按每个运行记录大模型输入、输出和总 Token，但用户真正需要的是跨全部 Agent 会话观察累计消耗，而不是在聊天流中逐次查看单一运行。现有记录也未保留供应商返回的缓存分项，且没有区分未来可计费的 VIP 调用与用户自带 API Key 调用，无法为统一用量中心和未来账户额度产品提供稳定的展示与契约基础。

## What Changes

- 复用现有运行级 `llm_usage.json` 作为原始数据，在后端按需聚合所有 Agent 会话的模型、调用次数、输入、输出、总 Token 以及供应商报告的缓存 Token。
- 新增顶层 `/usage` 用量中心，提供全部历史与日期筛选、每日趋势、Provider/模型分布，以及可展开的会话/运行明细。
- 从 Agent 聊天流移除单次用量卡片，保留运行详情中的单次运行用量，形成“全局汇总 + 运行追溯”的清晰层级。
- 扩展既有用量规范化和响应契约，以保留可用的缓存读写分项；供应商未报告时明确表示为未提供，而不进行估算。
- 为平台托管的 VIP 模型标注未来可计费用量，为账户、结算周期和余额摘要保留兼容的数据边界；用户自带 API Key 的调用保持本地可见但不可计费。
- 保持本地运行产物为观测与展示数据，而非充值、扣费、每日重置或余额裁决的权威账本。

## Capabilities

### New Capabilities

- `llm-usage-visibility`: 提供跨全部 Agent 会话的 LLM 用量汇总与运行级追溯，并规定缓存可用性与未来 VIP 计量兼容边界。

### Modified Capabilities

- 无。

## Impact

- 后端：`agent/src/agent/loop.py` 的用量规范化与持久化、运行详情 API，以及新的跨会话用量聚合服务和只读 API。
- 前端：新的 `/usage` 页面、侧栏入口、聚合 API 类型、图表、会话明细、运行详情展示、国际化和测试；Agent 聊天流不再呈现单次用量。
- 桌面登录与 VIP 配置：仅作为未来服务端计量身份关联的兼容边界；本次不修改认证、支付或配额执行逻辑。
