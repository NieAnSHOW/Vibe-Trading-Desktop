## Context

现有 AgentLoop 在每次模型调用完成后，从 `LLMResponse.usage_metadata` 累计输入、输出和总 Token，原子写入每个运行目录的 `llm_usage.json`，并向会话 SSE 发出 `llm_usage` 事件。运行详情 API 已能读取该文件。初版实现将用量接入 Agent 聊天流和运行详情，但用户确认目标应是独立的全局用量中心：按会话汇总全部历史运行，并保留运行详情作为单次追溯入口。

桌面登录会为平台托管 VIP 模型配置访问凭据，但本地运行产物既不能防篡改，也不能处理并发消费、充值、余额或结算。因此它只能承载可观测用量和未来服务端计量接口的兼容信息，不能成为账户额度的权威来源。

## Goals / Non-Goals

**Goals:**

- 在不增加代理、数据库或第二套采集管道的前提下，汇总所有 Agent 会话中的既有运行级用量。
- 提供独立的顶层用量页面，支持时间筛选、趋势、Provider/模型分布和按会话展开的运行明细。
- 在供应商报告缓存 Token 时保留其分项；缺失数据保持明确的“未提供”语义。
- 让每次运行明确区分平台 VIP 调用和用户自带 Key 调用，为后续服务端计量响应留出稳定字段和展示位置。
- 保持历史 `llm_usage.json` 和不报告 usage 的 Provider 可读、可用。

**Non-Goals:**

- 实现支付、充值、余额、每日重置、月度结算、服务端扣量或额度拦截。
- 对缺失的输入、输出或缓存 Token 做本地推测或估算。
- 将 API Key、登录 token、请求内容或响应内容写入用量产物、SSE 或页面。

## Decisions

### 1. 以 AgentLoop 为唯一采集点，复用现有文件和 SSE

AgentLoop 是获得聚合后的 Provider usage 元数据、run ID 和 session event callback 的唯一共同位置。扩展该处的规范化摘要可同时服务运行文件、SSE 和 `GET /runs/{run_id}`，避免 Tauri 代理或前端二次计数产生重复与偏差。

替代方案是在 Tauri 层代理模型 HTTP 请求。该方案无法可靠覆盖多 Provider/非标准传输，增加密钥处理风险，并会形成独立统计逻辑，故不采用。

### 2. 使用“已报告”语义，而非以零值表示未知缓存用量

输出契约将保留输入、输出、总量和调用数；缓存读取/写入仅在 Provider usage 元数据含有可识别的值时出现或标记为已报告。前端对缺失缓存字段显示未提供，避免“0”被误读为真实未命中。

替代方案是以字符数或总输入减去推算缓存量。该方案不具备 Provider 计费精度，且会污染将来的账单对账，故不采用。

### 3. 使用计量资格标记隔离未来 VIP 额度与本地统计

每个用量摘要将表达其来源 Provider 和 `metering_eligible` 语义：仅平台托管 `vip_server` 调用可标记为未来可计费；用户自带 Key 调用仍完整展示，但不可作为平台额度消耗。该标记是 UI 和未来计量服务的契约边界，而不是授权或扣费决定。

替代方案是通过 API Key 内容或本地用户 ID 判断。密钥不能暴露、可轮换且缺少可信账户身份；本地用户数据也不能充当支付身份，故不采用。

### 4. 使用后端按需聚合，不建立第二套用量索引

新增只读聚合服务，从 SessionStore 中读取会话与 attempts，以 `attempt.run_dir` 关联运行级摘要；旧会话关联不完整时，再从助手消息的 `metadata.run_id` 补充。相同 run ID 在一个或多个来源出现时只统计一次。该服务在页面打开或用户手动刷新时计算结果，不持久化新的全局账本。

替代方案是在每次调用后更新全局索引。它读取更快，但需要处理并发写入、历史重建、索引损坏和双写一致性，不符合本地桌面应用当前规模与只读观测目标。前端逐个请求运行详情则会产生大量请求并受列表上限影响，也不采用。

### 5. 使用顶层用量中心，并保留运行详情追溯

业务 WebUI 新增侧栏一级入口 `/usage`。页面默认展示全部历史，支持近 7 天、近 30 天、本月和自定义时间，显示总览、每日趋势、Provider/模型分布和按会话聚合的明细；会话可展开查看运行并链接到现有运行详情。页面只在首次进入和用户点击刷新时请求，不自动轮询。

Agent 聊天流移除实时与历史单次用量卡片，避免用量信息打断对话。Run Detail 保留运行级面板，便于从全局明细追溯异常运行。Tauri 控制台仍只管理依赖和 sidecar 生命周期，不承载业务统计。

### 6. 聚合筛选、缓存覆盖率与分页语义

聚合 API 接受可选起止时间、IANA 时区、会话搜索和分页参数，并使用 `[start_at, end_at)` 半开区间。时间优先使用 attempt 创建时间；消息 fallback 使用对应助手消息时间，两者都不可用时退回受约束目录修改时间。运行 ID 中的本地无时区时间不参与筛选。每日趋势按经验证的浏览器 IANA 时区分桶。

总览、趋势和 Provider/模型分布始终覆盖时间范围内的全部关联用量，不受会话搜索或分页影响。搜索只影响会话明细，匹配会话标题或 ID。缓存累计字段仅在至少一个运行报告该字段时出现，同时返回已报告和未报告运行数，避免将未知值误读为零。

## Risks / Trade-offs

- [不同 Provider 使用不同缓存字段名或不返回 usage] → 仅白名单、显式映射已知字段；缺失时显示未提供，并以单元测试覆盖。
- [历史产物缺少新字段] → 让 API 和前端字段保持可选，旧运行继续按已有总量展示。
- [本地数据被误用为账单] → 在数据模型、API 文档和 UI 中区分“运行统计”与未来“账户计量”；本次不暴露可扣费余额。
- [运行过程事件较频繁] → 复用现有每迭代一次的事件节奏和原子文件替换，不引入额外网络请求。
- [历史会话与运行关联不完整] → attempts 为主、助手消息 run ID 为补充，并按 run ID 去重；无法归属到任何会话的孤立运行不进入“所有会话”。
- [大量历史文件导致刷新变慢] → 采用用户触发的按需刷新和会话分页；保留未来替换为索引实现的服务边界，但本次不提前引入缓存数据库。
- [个别用量文件缺失或损坏] → 跳过单个运行并返回覆盖率/无效计数，页面展示部分数据提示，不让整个聚合失败。

## Migration Plan

1. 扩展后端 usage 规范化、摘要和响应模型，同时保留所有现有字段与文件路径。
2. 新增按需聚合服务与 API，覆盖会话关联、去重、日期筛选、时区分桶、分布统计和分页。
3. 新增 `/usage` 页面与导航入口，移除 Agent 聊天流的单次用量卡片，并保留 Run Detail 追溯。
4. 通过后端和前端测试验证历史兼容、聚合正确性、筛选/分页、错误降级、VIP 标记及密钥脱敏。
5. 若全局页面出现回归，可移除该页面与聚合端点而不影响既有运行文件、SSE 或模型调用；无需数据迁移。

## Open Questions

- 服务端计量 API、余额字段、每日重置时区和月付结算规则将在实际计费能力启动时另行设计；本 change 不定义其端点或持久化。
