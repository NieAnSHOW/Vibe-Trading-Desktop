## 背景

前端已有 `/watchlist` 路径、本地 SQLite 自选股列表、每 3 秒轮询的腾讯报价，以及将选中标的预填至 `/agent` 的能力。后端 `get_market_data` 和回测 loader 注册表则负责面向 Agent、研究、回测与风控的 Python 多源数据路由。

本变更新增一个面向 A 股研究工作流的 `/dashboard` 页面。`stock-sdk` 是浏览器与 Node.js 使用的 JavaScript SDK，能够以统一 TypeScript API 读取公共行情来源；它不提供交易级数据保证，也不应替代现有 Python 数据链路。

## 目标与非目标

**目标：**

- 提供市场优先的 A 股看板：指数概览、板块/概念脉冲、自选股、个股 K 线与 AI 盘面摘要。
- 将所有浏览器行情请求封装在可测试的 `stock-sdk` 适配层中，避免页面组件感知 SDK 返回形状。
- 以受鉴权、受限的 API 生成并缓存结构化 AI 摘要；数据与 AI 任一侧失败都不会阻断另一侧。
- 复用现有自选股持久化和 Agent 预填工作流。

**非目标：**

- 不改动 Python `get_market_data`、回测 loader 注册表、研究工具、live trading、风控或下单引用的数据源。
- 不将 `stock-sdk` 作为交易、风控、回测或事实核验的唯一数据来源。
- 不引入港股、美股、基金、期货或期权看板，不新增后台定时任务，也不持久化浏览器行情快照。
- 不接入 `stock-sdk` MCP server 或创建新的 Node.js 服务进程。

## 决策

### 浏览器端 `stock-sdk` 适配层

前端将固定依赖 `stock-sdk` v2，并在单一适配层中创建 `StockSDK` 实例，归一化为看板 DTO。看板页面和 Zustand store 只能依赖该适配层的领域方法，例如读取指数、自选股报价、板块排行和单个标的日 K。

这保留了用户选择的“直接使用 stock-sdk”，同时隔离 SDK API 变更、符号归一化和错误映射。页面不直接导入或调用 SDK；浏览器中不保存密钥，因为该 SDK 的本次用法不需要密钥。

备选方案是让 Python 后端或新增 Node 侧车代理 SDK。该方案会集中网络访问，但增加桌面打包和跨运行时复杂度，也违背本变更的浏览器直连范围，因此不采用。

### 市场优先看板与独立状态

新增懒加载 `/dashboard` 路由、侧边栏入口和中英文翻译。看板使用独立的 Zustand store 管理刷新、选择标的、局部错误和最近成功的数据；继续通过现有 watchlist API 读取本地自选股清单。

页面顶部展示上证、深证和创业板指数。主体按“市场脉冲 → 自选股 → 个股详情”排列：市场脉冲使用可用的板块/概念排行；自选股报价来自浏览器 SDK；选择一支股票后显示日 K、MA/MACD 等指标，并可沿用既有 Agent 预填跳转。

市场行情每 15 秒刷新；当 `document.visibilityState` 为 `hidden` 时暂停，恢复可见后立即刷新。每个数据区域独立保留最后成功值和时间，部分失败不会清空其他区域。

### 受限 AI 摘要 API

新增受现有本地/认证依赖保护的 `POST /dashboard/market-summary`。请求体是一个严格定义且有数量上限的 `MarketSnapshot`：市场标识、行情时间、3 个指数、有限数量的板块条目和有限数量的自选股条目。服务端拒绝超限数组、非数值字段、过大字符串和未支持市场。

该接口只使用已配置的 LLM 生成固定字段的盘面摘要，例如市场结论、驱动因素、风险提示与关注方向。它不能执行 Agent 工具、启动会话、读取交易账户或接受任意用户 prompt。响应包含 `market_as_of`、`generated_at`、摘要内容和 `stale` 状态。

服务端按市场维护 5 分钟内存缓存。正常页面行情轮询只复用有效摘要；用户明确请求刷新 AI 摘要时才跳过缓存。LLM 未配置或调用失败时返回可识别的可用性状态，前端仅降级 AI 卡片。

复用 `/sessions` SSE 工作流是备选方案，但会生成会话记录、污染聊天历史且无法稳定约束摘要结构与缓存，因此不采用。

### 可靠性、安全与可观测性边界

公共行情不具备交易级时效性。前端必须在每个区域展示最后成功的行情时间、来源标签和陈旧状态；使用超时/异常映射保留缓存数据。浏览器 CORS、SDK 上游限流或 SDK API 变动均被当作局部可恢复错误。

AI 摘要 API 只信任其经过 Pydantic 校验的结构，限制请求体规模，且不将原始快照写入数据库、日志或 Agent 会话。该摘要是研究辅助信息，任何交易或风控路径均不得读取它。

## 风险与取舍

- [公共上游延迟、限流或不可用] → `stock-sdk` 数据区域各自保留上次成功数据、显示时间和陈旧状态；不在 CI 访问真实接口。
- [浏览器 CORS 与网络环境差异] → 在本地开发服务器进行真实连通性验证；适配层将网络错误归一化并保留回退空间。
- [市场快照每 15 秒变化导致重复 LLM 调用] → AI 摘要按市场缓存 5 分钟，只有显式 AI 刷新绕过缓存。
- [来自浏览器的数据不可信或体积失控] → 后端使用白名单 DTO、字段/数组/字符串限制和认证依赖，且该数据不进入任何执行或交易路径。
- [新增 SDK 与现有 Python 数据不一致] → SDK 仅服务看板；Agent、回测、研究和风险系统继续使用既有源与其 fallback。

## 迁移计划

1. 添加并锁定前端 `stock-sdk` 依赖，实现受测适配层及 SDK DTO。
2. 新增看板状态、页面、路由、导航和翻译；复用现有自选股与 Agent 预填接口。
3. 添加 AI 摘要路由、输入/输出模型与内存缓存，并覆盖鉴权、缓存和失败降级。
4. 运行前端构建/Vitest、后端相关 pytest，并在开发服务器手工验证浏览器连通性。
5. 出现 SDK 兼容性或 CORS 问题时，移除看板入口和依赖即可回滚；既有 `/watchlist`、Agent 与 Python 数据路由不受影响。

## 待确认问题

无阻塞产品决策。实现阶段将通过浏览器连通性测试确认当前 `stock-sdk` 版本的 CORS 与所选 A 股 API 可用性；若失败，将在不触及 Agent 数据源的前提下回到设计阶段讨论替代的看板数据适配方式。
