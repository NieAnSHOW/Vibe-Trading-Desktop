# Brainstorm Summary

- Change: investment-news-hub
- Date: 2026-07-20
- Status: 用户已确认

## 确认的技术方案

- 在现有 FastAPI sidecar 内新增 `agent/src/news/` 原生领域包，分离来源配置、Pydantic 模型、采集器、LLM 摘要、快照存储和刷新协调器；`agent/src/api/news_routes.py` 提供 `/news-api/snapshot`、`/news-api/refresh`、`/news-api/refresh/status`。
- React 新增独立懒加载 `/news` 页面和轻量页面状态，不扩充 Agent 对话 store；Vite 代理 `/news-api`。桌面端使用可滚动赛道标签，窄屏使用选择菜单。
- 每次手动刷新覆盖全部 12 个赛道：202 快速返回，依次执行采集、标准化、赛道级新旧合并、LLM、校验和单快照原子提交。重复请求复用当前任务，退出取消未提交任务，重启回到 idle。
- 某赛道本次获得有效条目则更新；该赛道本次没有有效条目时沿用旧赛道并标记 stale，首次无旧数据则 unavailable。至少一个赛道更新才提交组合快照，全部无更新则任务失败且不写盘。
- 每次刷新通过 `build_llm()` 创建一个当前 Provider 实例；每个有新数据的赛道使用一次结构化调用，同时返回标题映射和 3–5 条要点，候选最多 16 条，总并发上限 3。单赛道 LLM 失败只降级该赛道。
- 来源并发 16，连接/读取超时 5/15 秒，单 feed 2 MiB，最多 3 次合法重定向；每源最多 6 条、最近 7 天，每赛道快照最多 100 条；标题/摘要最多 300/1000 字符。
- 使用安全 XML 解析、HTTP(S) URL 与重定向校验、纯文本规范化、React 文本渲染、稳定错误码和敏感信息清理。常规流程不抓取媒体全文。

## 关键取舍与风险

- 选择原生模块而非脚本适配或第二 sidecar，以统一鉴权、LLM、打包和错误模型；代价是需要重写上游服务与 UI 层。
- 选择一次刷新全部赛道，保持单一操作与任务模型；代价是延迟和 LLM 调用高于单赛道刷新。
- 选择赛道级旧数据回退，避免局部故障清空内容；代价是组合快照中的赛道可能具有不同 freshness。
- 选择每赛道一次 LLM 合并调用，控制调用量并自然隔离赛道故障；代价是单赛道标题与要点共享一次失败域。
- 固定上游完整 SHA 和 golden manifest/hash，不运行时跟随上游；来源变化需要代码审查和发布。

## 测试策略

- 领域层使用离线 RSS/Atom fixtures，覆盖 DTD/XXE、协议、字段边界、过滤去重、部分失败和赛道回退。
- LLM 完全 mock `build_llm()`，覆盖结构输出、中文标题校验、提示注入、单赛道失败和并发上限。
- 存储使用临时目录和故障注入，覆盖 0600、损坏文件、fsync、原子替换和旧快照保留。
- API 覆盖 202 复用、鉴权、SPA fallback、空状态、稳定错误码和脱敏；前端覆盖路由、响应式赛道切换、轮询生命周期、降级、链接安全和 i18n。
- 常规测试不访问真实 RSS 或真实 LLM；真实 feed 仅人工冒烟。验证 MIT notice 进入源码、sdist/wheel 和桌面 assemble 资源；新增解析依赖时运行 Tier 0 smoke。

## Spec Patch

- 将 proposal 中“保存当前刷新任务状态”的表述收紧为“任务状态仅驻内存”。
- 增加一次刷新全部 12 个赛道的场景，并修正“至少一个来源成功即可提交”的过宽判定。
- 增加赛道级陈旧回退、首次无旧数据时 unavailable、全部赛道无更新时整次失败，以及每赛道 `generated_at`/`stale` 的验收场景。
- 明确每赛道一次结构化 LLM 调用、最多 16 条候选、并发上限 3；将高层 design 中“每次调用都 build_llm”修正为“每次刷新构建一个实例”。
